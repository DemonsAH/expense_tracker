# Expense Tracker LangChain 化产品需求文档

## 1. 项目概述

本项目目标是构建一套面向个人与多人共同消费场景的小票智能处理系统。系统从本地固定目录读取小票图片，调用多模态模型直接输出结构化消费数据，经 LangChain 工作流完成校验、重试、归档、人工审核、落库和报表生成。

系统当前目标能力包括：

- 小票图片批量处理
- 多模态模型直出固定结构 JSON
- 商品级消费结构化
- 归属人自动识别
- 自动重试与失败归档
- 人工审核失败小票
- 月度 / 季度消费统计
- 品类消费统计
- 商品价格涨跌分析
- 每月自动生成报表
- 面向非技术用户的 `exe` 图形界面启动方式
- 开发阶段可通过 CLI 触发完整主链

本需求文档仅保留当前目标方案，不再保留旧版方案描述。

## 2. 项目目标

项目需要实现一套可持续运行的小票处理与消费分析流程：

1. 从固定目录或触发入口接收小票图片。
2. 将图片保存到本地固定目录，保留原始文件。
3. 调用多模态模型识别小票并直接返回目标数据结构。
4. 对结果进行结构与业务校验。
5. 校验通过后写入本地结构化存储。
6. 校验失败后进入重试、归档与人工审核流程。
7. 按月自动生成消费报表。
8. 在图形界面中支持查看、修正和复核已有小票数据。

## 3. 输入与触发方式

### 3.1 本地目录输入

系统需要支持从预设本地目录批量读取小票图片。

要求包括：

- 支持 JPG、JPEG、PNG、WEBP 等常见图片格式
- 支持批量处理整个目录中的图片
- 支持记录每张图片的处理结果
- 支持为每次模型调用保存原始返回内容，便于排查问题
- 文件名中建议包含时间戳、原始文件名、模型名、尝试次数

### 3.2 手机消息触发小票分析

系统需要预留“手机消息触发小票分析”能力。具体触发方式后续确定，但需求先明确如下：

- 手机端触发后，系统能够接收到小票照片文件
- 触发后必须先将小票照片保存到本地固定目录
- 保存成功后再进入识别流程
- 保存的文件需具备唯一文件名，避免覆盖
- 保存失败时不得继续调用模型
- 后续所有识别、校验、归档、人工审核流程与本地目录导入保持一致

说明：

- 当前阶段只明确触发后的系统行为，不限定消息平台、协议或接入方式
- WhatsApp 等外部触发接入层后续应优先调用 Python 主链函数，而不是依赖 shell 解析 CLI 输出
- 本地固定目录路径应可配置

### 3.3 定时任务触发

系统需要支持定时任务：

- 定时扫描待处理目录中的新图片
- 每月自动生成报表

每月自动报表要求：

- 按自然月生成
- 默认在每月结束后自动生成上一个月的报表
- 报表生成结果需落地保存
- 支持后续在界面中查看历史月报

## 4. LangChain 工作流需求

### 4.1 工作流职责

LangChain 工作流需要承担以下职责：

- 图片输入管理
- 归属人配置加载
- Prompt 模板管理
- 多模态模型调用
- 结构化输出约束
- 结果校验
- 重试控制
- 失败归档
- 数据落库
- 报表触发
- LangSmith 可选 tracing

### 4.2 建议工作流节点

建议工作流至少包含以下节点：

1. 图片接收/读取节点
2. 本地文件保存节点
3. 归属人配置加载节点
4. Prompt 组装节点
5. 多模态模型调用节点
6. 结构化 JSON 解析节点
7. 业务校验节点
8. 重试决策节点
9. 后处理节点
10. 失败归档节点
11. 数据存储节点
12. 月报生成节点

### 4.3 处理主链路

主链路应为：

1. 接收小票图片
2. 保存原图到本地固定目录
3. 加载归属人配置与系统配置
4. 调用多模态模型获取结构化结果
5. 校验结构、金额、品类、归属人和业务规则
6. 对取消项和特殊负数项做后处理
7. 通过则写入正式数据
8. 不通过则自动重试
9. 重试仍失败则归档并进入人工审核池

### 4.4 开发测试入口

开发阶段需要提供 CLI 入口，至少支持单张图片跑通完整主链，便于：

- 本地调试
- LangSmith 观察
- 回归测试
- 快速验证 Prompt 或模型变更

## 5. 模型输出要求

### 5.1 输出原则

模型不再返回自由文本说明，不再以 OCR 原文 + Python parser 作为主解析方式。模型必须直接返回固定结构 JSON，供程序校验和落库。

输出要求：

- 只返回 JSON
- 不返回 markdown 代码块
- 不返回解释、备注、分析过程
- 字段名固定
- 字段类型固定
- 品类值必须从预设枚举中选择
- 归属人字段必须引用配置中的合法 `owner_id`

### 5.2 输出结构

模型输出的顶层结构定义如下：

```json
{
  "merchant": "string",
  "purchase_date": "YYYY-MM-DD",
  "currency": "EUR",
  "total_amount": 0.0,
  "payment_method": "string or null",
  "owner_mode": "normal | receipt_owner | item_owner",
  "default_owner_id": "string",
  "receipt_owner_marker": "string or null",
  "items": [
    {
      "name": "string",
      "normalized_name": "string",
      "category": "SNACKS | PERSONAL_CARE | HOUSEHOLD | DRINK | MEAT | VEGGIE | FRUIT | OTHER | DINING",
      "quantity": 0.0,
      "unit_price": 0.0,
      "total_price": 0.0,
      "owner_id": "string",
      "owner_marker": "string or null"
    }
  ]
}
```

字段约束：

- `purchase_date` 使用 `YYYY-MM-DD`
- `currency` 当前阶段固定为 `EUR`
- `quantity`、`unit_price`、`total_price` 使用数字
- `default_owner_id`、`owner_id` 必须能在归属人配置中找到
- 模型输出层允许出现合法负数项，后续由业务校验与后处理决定其归属

### 5.3 商品品类枚举

商品品类固定为以下枚举：

- `SNACKS`
- `PERSONAL_CARE`
- `HOUSEHOLD`
- `DRINK`
- `MEAT`
- `VEGGIE`
- `FRUIT`
- `OTHER`
- `DINING`

规则说明：

- `DINING` 不参与商品价格涨跌排行
- 其余品类参与正常统计与分析
- 模型输出非法品类时视为校验失败

### 5.4 精炼 Prompt

系统需内置一版精炼 Prompt，重点保证结构稳定、信息干扰最小。

推荐 Prompt 如下：

```text
你将看到一张购物小票图片。请直接提取并返回一个 JSON 对象。

要求：
1. 只输出 JSON，不要输出任何解释、注释或代码块。
2. 所有金额、数量使用数字。
3. 日期格式必须是 YYYY-MM-DD。
4. currency 固定为 EUR。
5. owner_mode 只能是：normal, receipt_owner, item_owner。
6. category 只能是：SNACKS, PERSONAL_CARE, HOUSEHOLD, DRINK, MEAT, VEGGIE, FRUIT, OTHER, DINING。
7. 识别整单归属标记和商品行末归属标记。
8. 若无法确定 payment_method 或 owner_marker，可填 null。
9. 输出必须符合固定 JSON 结构。
```

说明：

- 正式实现中可在该 Prompt 外层补充归属人配置、合法 `owner_id` 列表等动态上下文
- 核心 Prompt 应尽量保持短、稳定、结构明确

## 6. 核心业务规则

### 6.1 商品级结构化

系统的核心分析粒度为商品 item，而不是整张小票。

每个商品至少需要包含：

- 商品名称
- 标准化商品名称
- 品类
- 数量
- 单价
- 总价
- 归属人
- 归属人标记

说明：

- 对于称重商品，重量直接作为 `quantity`
- `unit_price` 为单位价格
- `total_price` 为该商品最终记账金额

### 6.2 取消项、负数项与 Leergut 处理

模型与工作流需支持识别取消项和合法负数项，例如：

- `Storno`
- `Sofortstorno`
- `Rücknahme`
- `Leergut`
- `Pfand`

处理规则如下：

- 全部条目（包括 Storno/Sofortstorno 取消项、Leergut/Pfand 负数项）均保留在正式商品列表 `items` 中
- 取消项和被取消原商品前后两条记录都保留，用户可自行查看原始数据
- 负数项（包括取消项和 Leergut/Pfand）的 `total_price` 为负值，参与月度支出总和时自然抵消
- `Leergut/Pfand` 项参与每月支出总和

说明：

- 取消项不再作为特殊处理流程，不触发自动移除逻辑

### 6.3 正式数据与排行规则

系统需要区分“正式消费数据”和“排行分析数据”：

- 正式消费数据用于月度总支出、归属人支出、品类统计、落库
- 价格排行数据用于涨价/降价分析

规则如下：

- 所有条目（包括取消项和 Leergut/Pfand）均属于正式消费数据
- `DINING` 不进入价格排行
- `Leergut/Pfand` 不进入价格排行
- 所有负数项（包括取消项和 Leergut/Pfand）不进入价格排行

### 6.4 归属人识别

系统支持消费归属人识别，并允许用户通过 JSON 配置归属人列表。

归属人配置文件中每个人需包含：

- `id`
- `name`
- `marker`
- `is_me`

系统需支持三种归属模式自动识别：

- `normal`
- `receipt_owner`
- `item_owner`

#### `normal`

- 不识别任何归属人标记
- 所有商品默认归属 `Me`

#### `receipt_owner`

- 识别整张小票顶部的归属人标记
- 整张小票商品默认归属于该归属人

#### `item_owner`

- 识别商品行末尾的归属人标记
- 行级标记优先
- 未标记商品回退到整单归属
- 若整单也未标记，则回退到 `Me`

### 6.5 手写标记规范

为支持 OCR 后自动识别归属人，约定如下：

- 整张小票归属人标记：用户手写在小票右上角空白处
- 推荐格式：`@M`、`@A`、`@B`
- 单个商品归属人标记：用户手写在商品行末尾
- 推荐格式：`M`、`A`、`B`

系统应允许归属人识别功能开关可控：

- 可开启归属人识别
- 可关闭归属人识别
- 关闭时统一进入 `normal` 模式，并默认归属 `Me`

## 7. 校验、重试与归档

### 7.1 结果校验

虽然模型直接输出结构化 JSON，但系统仍需要做业务校验。

至少包括以下校验：

- JSON 结构完整且字段类型合法
- `default_owner_id` 存在于 `owners.json`
- 每个 `item.owner_id` 存在于 `owners.json`
- `quantity * unit_price` 与 `total_price` 在允许误差范围内一致
- 小票 `total_amount` 与商品金额汇总在允许误差范围内一致
- 类别值必须属于合法枚举
- 日期格式合法

### 7.2 自动重试

重试规则如下：

- 先请求一次模型
- 若结果不理想且属于可重试错误，则自动重试
- 最多重试 3 次
- 若 3 次都失败，则判定该小票暂不录入
- 每次尝试都需保留模型返回文本

建议作为重试判据的错误包括：

- 非法 JSON
- 不符合目标 schema
- 归属人 ID 不合法
- 金额校验失败
- 负数项不合法

### 7.3 失败归档

对无法通过校验的小票，系统需自动归档。

归档要求：

- 保存原始图片
- 保存每次模型返回内容
- 保存失败原因
- 保存尝试次数
- 保存时间戳

该目录用于后续人工复查、标注与 Prompt 优化。

## 8. 人工审核与图形界面需求

### 8.1 `exe` 启动后的主界面

系统打包为 `exe` 后，启动应直接进入图形界面。

主界面至少需要提供以下能力：

- 启动小票处理任务
- 查看当前待处理目录
- 查看处理成功的小票数量
- 查看识别失败待审核的小票数量
- 查看最新月报生成状态

### 8.2 失败小票人工审核

系统必须提供人工审核识别不通过的小票界面。

审核界面至少包括：

- 查看失败小票原图
- 查看模型原始输出
- 查看校验失败原因
- 查看系统建议修正点
- 手工修改结构化字段
- 重新提交校验
- 审核通过后写入正式数据

### 8.3 已有小票查看与修改

系统必须提供查看现有小票详情和修改接口。

要求包括：

- 查看历史小票列表
- 查看单张小票原图
- 查看小票级字段
- 查看商品级字段
- 查看被移除的取消项审计记录
- 支持修改商品名称、标准化名称、品类、数量、单价、总价、归属人
- 支持修改整单归属模式和默认归属人
- 支持保存修改记录

### 8.4 审核与修改原则

人工修改后的数据仍需经过系统校验。

要求包括：

- 修改后重新校验金额与枚举合法性
- 审核通过的数据应标记为人工确认
- 保留原始模型输出与人工修改后的最终版本

## 9. 数据模型需求

### 9.1 Receipt

建议保留如下字段：

- `id`
- `merchant`
- `purchase_date`
- `currency`
- `total_amount`
- `payment_method`
- `default_owner_id`
- `owner_mode`
- `receipt_owner_marker`
- `ocr_raw_text`
- `is_verified`
- `ocr_status`
- `ocr_attempts`
- `ocr_failure_reason`
- `image_path`
- `items`
- `removed_items`

说明：

- `items` 为正式消费数据
- `removed_items` 为审计记录（当前阶段无自动移除项，保留字段以备后续使用）

### 9.2 ReceiptItem

正式商品建议保留如下字段：

- `id`
- `receipt_id`
- `name`
- `normalized_name`
- `category`
- `quantity`
- `unit_price`
- `total_price`
- `owner_id`
- `owner_marker`

### 9.3 Removed Item

被移除审计项建议保留如下字段：

- `name`
- `normalized_name`
- `category`
- `quantity`
- `unit_price`
- `total_price`
- `owner_id`
- `owner_marker`
- `reason`
- `related_index`

### 9.4 Owner JSON

示例结构如下：

```json
{
  "owners": [
    { "id": "me", "name": "Me", "marker": "M", "is_me": true },
    { "id": "alice", "name": "Alice", "marker": "A", "is_me": false },
    { "id": "bob", "name": "Bob", "marker": "B", "is_me": false }
  ]
}
```

约束要求：

- `id` 不可重复
- `marker` 不可重复
- `marker` 使用单个大写字母
- `is_me` 必须且只能有一个为 `true`

### 9.5 本地持久化格式

当前阶段推荐使用 JSON 作为主存储格式。

原因：

- 数据结构包含嵌套 `items`、`removed_items`、失败归档和审核字段
- 与模型输出 JSON 和 `pydantic` schema 更自然匹配
- 更适合快速迭代字段结构

CSV 可作为后续导出格式，不建议作为主存储格式。

## 10. 分析与报表需求

### 10.1 我的支出

报表需支持：

- 我的每月总支出
- 我的每季度总支出
- 我的同比增长
- 我的环比增长

### 10.2 其他归属人支出

报表需支持：

- 除 `Me` 之外每个归属人的每月总支出
- 各归属人的同比增长
- 各归属人的环比增长

### 10.3 品类支出

报表需支持：

- 各品类总支出
- 各品类月度 / 季度支出变化
- 品类占比分析

### 10.4 商品价格分析

系统需支持比较某个商品是否涨价，并给出每月价格涨跌排行。

要求包括：

- 识别同一商品的历史价格变化
- 支持按月生成涨价排行
- 支持按月生成降价排行
- 不包含 `DINING` 品类
- 不包含取消项和被取消商品
- 不包含 `Leergut/Pfand`

为了支撑价格分析，建议保留：

- `normalized_name`
- `unit_price`
- `purchase_date`

### 10.5 每月自动报表

系统需支持每月自动生成报表。

至少包括：

- 月总支出
- 归属人维度支出
- 品类维度支出
- 商品涨价排行
- 商品降价排行

输出要求：

- 报表可保存为本地文件
- 报表命名包含年月
- 支持在图形界面中查看和打开

## 13. 验收标准

系统验收至少满足以下标准：

- 能从预设目录读取小票图片
- 能接收手机消息触发后保存到本地固定目录的小票图片
- 能在保存成功后调用多模态模型直接返回目标 JSON
- 模型输出字段结构固定，且仅输出 JSON
- 内置精炼 Prompt，并可被配置管理
- 不依赖 Python 规则 parser 作为主解析流程
- 能正确识别商品名称、数量、单价、总价、品类
- 品类枚举为 `SNACKS`、`PERSONAL_CARE`、`HOUSEHOLD`、`DRINK`、`MEAT`、`VEGGIE`、`FRUIT`、`OTHER`、`DINING`
- `DINING` 不参与价格涨跌排行
- 能支持归属人 JSON 配置
- 能支持 `normal / receipt_owner / item_owner` 三种模式
- 能识别 `@M` 等整单标记和行尾字母标记
- 能识别称重商品
- 能处理 `Sofortstorno` 等取消项
- 合法识别 `Leergut/Pfand` 等负数项
- `Leergut/Pfand` 计入月度总支出，但不进入价格排行
- 负数取消项及被取消原商品均保留在正式商品数据中
- 校验失败时最多重试 3 次
- 3 次失败后进入失败归档目录
- 每次模型输出都可保存留档
- 正式结果可写入本地 JSON store
- CLI 可作为开发测试入口启动完整主链
- `exe` 启动后直接进入图形界面
- 图形界面可人工审核识别失败的小票
- 图形界面可查看并修改已有小票数据
- 能自动生成月报并保存
- 提供可交付给非技术用户使用的启动方式
